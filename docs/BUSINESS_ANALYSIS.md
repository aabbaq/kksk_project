# KKSK 博客 — 当前业务功能分析与改进建议

> 最后更新：2026-06-06  
> 关联文档：[REFACTOR_PLAN.md](./REFACTOR_PLAN.md)

---

## 1. 产品定位

**KKSK / Lothric Castle** 是一个带权限分级的个人博客系统。核心卖点是：

- Markdown 写作与渲染
- 基于 `userrole`（用户角色等级）和 `secretLevel`（文章密级）的**分级可见性**
- 文章可设为隐藏（hidden）、密码保护（protected）
- 深色主题、卡片式首页、偏个性化的 UI 文案

目标用户：博主本人 + 不同权限等级的读者（游客、普通用户、管理员）。

---

## 2. 用户与角色体系

### 2.1 用户模型（MongoDB `users` 集合）

| 字段 | 说明 |
|------|------|
| `username` | 登录名 |
| `nickname` | 昵称 |
| `password` | MD5 哈希密码 |
| `userrole` | 角色等级（数字，如 0=游客逻辑、7=管理员） |
| `registerDate` | 注册时间 |
| `biography` | 个人简介 |
| `alias` | 别名 |
| `emoji` | 表情标识 |

### 2.2 角色权限（`userrole`）

| 角色值 | 含义 | 能力 |
|--------|------|------|
| `0` | 游客（未登录逻辑） | 仅看 `secretLevel=1` 且 `hidden=false` 的文章 |
| `1~6` | 普通登录用户 | 看自己的文章 + 密级 ≤ 自身角色的公开文章 |
| `7` | 管理员 | 看全部文章、删除任意文章 |

### 2.3 认证流程

1. 用户在 `/login` 输入用户名密码
2. 后端 MD5 校验，签发 JWT（有效期 3 小时）
3. Token 存入 `sessionStorage`，Axios 拦截器自动附加 `Authorization: Bearer <token>`
4. 访问需登录页面时，路由守卫检查 Token，并调用 `/api/user/tokenCheck` 校验
5. 用户中心可退出登录（清除 sessionStorage）

**缺失**：无用户注册、无密码找回、无刷新 Token 机制。

## 3. 文章（博客）业务

### 3.1 文章模型（MongoDB `texts` 集合）

| 字段 | 说明 |
|------|------|
| `number` | 文章序号（自增） |
| `title` / `subtitle` | 标题 / 副标题 |
| `author` / `owner` | 作者名 / 所有者 ID |
| `tag` | 标签（仅存储，无筛选功能） |
| `picture` | 封面图文件名（对应 `static/*.jpg`） |
| `content` | Markdown 原文 |
| `htmlContent` | 渲染后的 HTML |
| `secretLevel` | 密级 1~3（N / S / D） |
| `protected` | 是否密码保护 |
| `protectedPassword` | 保护密码（明文存储） |
| `hidden` | 是否隐藏（仅授权用户可见） |
| `date` / `lastDate` | 创建 / 最后修改时间 |

### 3.2 文章可见性规则

```
管理员 (role=7) 或文章 owner → 始终可见

其他用户：
  - hidden=true → 不可见
  - hidden=false → 需 userrole >= secretLevel
```

### 3.3 文章 CRUD

| 操作 | 入口 | 权限 |
|------|------|------|
| 列表浏览 | 首页 `/homepage` | 按角色过滤 |
| 详情阅读 | `/content/:author/:title` | 按可见性规则 |
| 新建 | 用户中心 Texts → Post | 需登录 |
| 编辑 | 首页编辑按钮 / 用户中心 Edit | 需登录 |
| 删除 | 首页 / 用户中心 / 浮动按钮 | 仅 `userrole=7` |

### 3.4 写作流程

1. 填写标题、副标题、标签、封面图名
2. 设置密级滑块（1~3）、隐藏开关、密码保护开关
3. Markdown 编辑器（vue-simplemde）写作
4. 提交时前端用 `marked` 转 HTML，一并上传
5. 新建或更新（`blogupdate` 标志区分）

## 4. 页面与功能清单

### 4.1 已实现且可用的功能

| 页面/模块 | 路由 | 功能 |
|-----------|------|------|
| 首页 | `/homepage` | 文章卡片列表、跳转详情、登录后显示编辑/删除 |
| 登录 | `/login` | 用户名密码登录 |
| 文章详情 | `/content/:author/:title` | 封面图、Markdown HTML 渲染 |
| 文章编辑 | `/post/:textNumber` | Markdown 写作、密级/隐藏/保护设置 |
| 用户中心-个人 | `/user/myself` | 展示用户信息、退出登录 |
| 用户中心-文章 | `/user/texts` | 文章卡片管理（查看/编辑/删除） |
| 顶栏 | 全局 | 品牌导航、登录/用户入口、用户 Tab |
| 浮动按钮 | 全局 | 回到顶部；详情页可编辑/删除 |
| 刷新中转 | `/refresh` | 删除后刷新列表的 hack 页 |

### 4.2 半成品 / 占位功能

| 页面/模块 | 现状 | 问题 |
|-----------|------|------|
| 用户中心-Others | 仅显示 "Others" 文字 | 无业务逻辑 |
| 用户中心-News | 仅显示 "News" 文字 | 无业务逻辑 |
| 修改昵称 | UI 有编辑按钮 | `changeNickname()` 仅切换 readonly，无 API |
| 修改密码 | UI 有编辑按钮 | `changePassword()` 仅 console.log，无 API |
| 修改简介 | UI 可编辑 | 无保存 API |
| Share / Explore 按钮 | UserSelf 卡片上 | 无实现 |
| 标签 `tag` | 编辑时可填 | 无列表筛选、无标签页 |
| 受保护文章密码 | 编辑时可设 | 阅读时无密码验证流程 |
| 文章详情删除 | 代码已注释 | 功能被禁用 |

### 4.3 已知技术债务（影响业务）

| 问题 | 影响 |
|------|------|
| 前后端 API 路径不一致 | 部分请求 404（如 `/api/login` vs `/api/user/login`） |
| `generateInfo` 未定义 | 登录、获取用户信息接口可能报错 |
| `mongodb.js` 查询用回调但未正确 return | 登录等数据库操作可能失效 |
| JWT `decrypt` 返回结构与路由使用不匹配 | Token 校验后取不到 `id` |
| 登录页预填测试账号密码 | 安全隐患 |
| `protectedPassword` 明文存储与前端展示 | 安全风险 |
| `Refresh.vue` 中转刷新 | 体验差，应改为组件内刷新 |

## 5. 功能建议

### 5.1 建议删除

| 功能 | 理由 |
|------|------|
| **UserOthers 页面** | 纯占位，无业务定义；重构时移除，有需求再设计 |
| **UserNews 页面** | 纯占位；若不做站内通知/动态流，不应保留空 Tab |
| **UserSelf 的 Share / Explore 按钮** | 无实现、无规划，属于 UI 噪音 |
| **Login 预填账号密码** | 安全风险，必须删除 |
| **Refresh 中转页** | 技术 hack，改为列表页 `reload` 或 Pinia 状态刷新 |
| **EJS 渲染首页**（`backend/rapp.js`） | 前后端分离后由 Vite 构建托管，EJS 可移除 |
| **`path` npm 包** | Node 内置模块，不应作为依赖 |
| **冗余 Markdown 样式文件** | `onigiri*.css`、`purple.css` 等多套未统一样式，保留一套即可 |
| **明文展示 protectedPassword**（UserTexts） | 安全风险，必须删除该展示逻辑 |

### 5.2 建议保留并完善

| 功能 | 完善方向 |
|------|----------|
| **文章 CRUD** | 修复 API，统一路径；非管理员应只能删自己的文章 |
| **分级可见性（secretLevel + hidden）** | 核心特色，保留；简化为「公开 / 登录可见 / 指定角色」更易理解 |
| **Markdown 编辑与渲染** | 换 md-editor-v3；服务端也应做 HTML 消毒（防 XSS） |
| **封面图** | 保留；改为上传接口，而非手动填文件名 |
| **用户中心个人页** | 保留；补齐资料修改、密码修改 API |
| **JWT 认证** | 保留；增加 refresh token 或滑动过期 |
| **顶栏 + 浮动按钮** | 保留；精简逻辑，按路由配置操作按钮 |
| **卡片式首页** | 保留；增加分页或无限滚动 |

### 5.3 建议新增

| 功能 | 优先级 | 说明 |
|------|--------|------|
| **用户注册** | 高 | 当前只能数据库手动建用户 |
| **修改密码 / 资料 API** | 高 | 与现有 UI 对齐 |
| **受保护文章阅读密码验证** | 高 | 补齐 protected 闭环 |
| **图片上传** | 高 | 替代手写 `picture` 文件名 |
| **标签筛选 / 标签页** | 中 | `tag` 字段已有，补 UI 即可 |
| **文章搜索** | 中 | 按标题、标签、作者 |
| **分页** | 中 | 文章增多后必须 |
| **草稿箱** | 中 | 编辑中途保存 |
| **评论系统** | 低 | 按需；可接第三方 |
| **RSS / Sitemap** | 低 | 博客标配，利于 SEO |
| **管理后台** | 低 | 用户管理、文章审核（若多人使用） |
| **操作日志 / 审计** | 低 | 删除等敏感操作留痕 |

### 5.4 建议简化的复杂设计

| 现有设计 | 建议 |
|----------|------|
| `secretLevel` 1~3 + `hidden` + `protected` 三套叠加 | 合并为「可见性」枚举：`public` / `members` / `private` / `password`，降低用户认知成本 |
| `userrole` 数字等级 0~7 | 改为明确角色：`guest` / `user` / `admin`，密级与角色解耦 |
| 前后端双份 Markdown 渲染 | 后端统一渲染并消毒，前端编辑器只做预览 |

## 6. 重构后的功能范围建议（MVP）

### 第一期（必须有）

- 用户：注册、登录、退出、修改资料、修改密码
- 文章：列表、详情、创建、编辑、删除（所有者 + 管理员）
- 权限：公开 / 登录可见 / 私密 / 密码保护
- Markdown 编辑与安全渲染
- 封面图上传

### 第二期（增强）

- 标签筛选
- 搜索
- 分页
- 草稿

### 第三期（可选）

- 评论
- 站内通知（替代原 News Tab）
- RSS / SEO
- 管理后台

---

## 7. 总结

| 维度 | 现状 | 建议 |
|------|------|------|
| **核心价值** | 带密级的个人 Markdown 博客 | 保留，简化权限模型 |
| **可用功能** | 登录、文章 CRUD、分级浏览 | 修复 bug 后作为 MVP 基础 |
| **应删除** | 占位页、假按钮、预填密码、Refresh hack、明文密码展示 | 减少噪音与风险 |
| **应补充** | 注册、资料修改、密码验证阅读、图片上传、分页搜索 | 补齐博客闭环 |
| **技术债务** | API 不一致、后端 bug、无 XSS 防护 | 重构时优先修复 |

当前项目**业务意图清晰**（私密分级博客），但**大量 UI 超前于后端实现**，且存在多处接口与安全问题。重构时应「先收敛功能、再扩展」，以第一期 MVP 为目标，避免把占位页面原样迁移到新架构。
